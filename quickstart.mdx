---
title: "Get started with BuilderPulse"
description: "Subscribe to BuilderPulse daily reports in under 5 minutes using GitHub Watch, RSS, or direct links to English and Chinese editions."
---

BuilderPulse requires no account, no email address, and no subscription form. Reports live in a public GitHub repository and are delivered via standard mechanisms — GitHub notifications and RSS — that you already have access to. Follow the steps below to start receiving daily reports and get the most out of each one.

<Steps>
  <Step title="Watch the GitHub repository">
    Go to [github.com/BuilderPulse/BuilderPulse](https://github.com/BuilderPulse/BuilderPulse) and click **Watch** in the top-right corner. From the dropdown, select **All Activity** to receive a GitHub notification every time a new report is published.

    You can also click **Star** to bookmark the repository and make it easier to find later — but Watch is what triggers notifications.

    <Info>
      Each new daily report is committed directly to the main branch. GitHub sends a notification for every commit when you watch with All Activity enabled.
    </Info>
  </Step>

  <Step title="Subscribe via RSS">
    Add the following Atom feed URL to your RSS reader:

    ```
    https://github.com/BuilderPulse/BuilderPulse/commits/main.atom
    ```

    This feed updates every time a new commit lands on the main branch — which happens once per day when the new report publishes. It works with any RSS reader that supports Atom feeds, including Feedly, NetNewsWire, Reeder, and Miniflux.

    <Tip>
      If you already use an RSS reader for other developer news (Hacker News, GitHub release feeds, blog posts), adding BuilderPulse to the same reader keeps your reading workflow in one place.
    </Tip>
  </Step>

  <Step title="Read your first report">
    Reports are stored at a predictable path in the repository. To read today's English report, navigate to:

    ```
    en/YYYY/YYYY-MM-DD.md
    ```

    For example, the April 16, 2026 report is at `en/2026/2026-04-16.md`. You can read it directly on GitHub or clone the repository to read locally.

    The README at the root of the repository always links to the most recent report — so if you're not sure what today's date is in the report schedule, start there.

    <Tip>
      A Chinese edition is published at the same time at `zh/YYYY/YYYY-MM-DD.md`. The Chinese edition is written for Chinese-speaking builders — not a machine translation — and sometimes includes different framing or additional context relevant to the Chinese tech ecosystem.
    </Tip>
  </Step>

  <Step title="Navigate the report sections">
    Every report follows the same five-section structure. Use the section headers to jump to what's most relevant to you:

    - **Discovery** — What launched today, which search terms are surging, which GitHub projects have no commercial version yet, and what developers are complaining about. Start here if you're looking for a build idea.
    - **Tech Radar** — Product shutdowns and downgrades, fastest-growing developer tools, and new AI model releases with consumer product analysis. Start here if you're tracking the competitive landscape.
    - **Competitive Intel** — Real revenue numbers from indie developers, dormant projects reviving, and migration stories. Start here if you're thinking about pricing or positioning.
    - **Trends** — Keyword frequency shifts, VC focus areas, and the new-word radar. Start here if you're planning content or SEO strategy.
    - **Action** — The concrete build recommendation. This section always gives you a 2-hour project and a weekend expansion, with explicit reasoning for why the data supports each one. Start here if you have limited time and want to act on the day's findings directly.

    <Note>
      The **Action** section at the end of each report also includes the most counter-intuitive finding of the day. These are often the most useful: the April 13, 2026 report's counter-intuitive finding was that Berkeley researchers had gamed all 8 mainstream AI agent benchmarks to near-perfect scores without solving any actual tasks — which changes how you should evaluate model selection entirely.
    </Note>
  </Step>
</Steps>
